# **从 Nexus 到 kkRepo：我们公司的制品仓库平替实践**

## 前言
博主评语：以下来自真实的 kkRepo 用户以第一人称视角所记录的在一家光伏制造企业的落地案例。从 Nexus 迁移到 kkRepo（https://github.com/klboke/kkRepo） 只用了一天的时间，就获得了高可用的制品仓库基建。

## **背景**

我们是一家光伏制造企业，研发团队超过 500 人，覆盖 50 多个项目，内部多条产品线共用统一的 CI/CD 流水线。制品仓库作为构建基础设施的核心组件，承担着 Maven、npm、pypi等多种制品的托管与分发职责，是整个研发体系的关键基础设施。

过去几年，我们一直使用 Sonatype Nexus Repository `3.78.2-04`（Community Edition），通过 Docker Compose 部署，底层数据库使用 PostgreSQL，制品 blob 存储使用 OSS。随着业务规模扩大，Nexus Community Edition 的一些限制逐渐暴露出来，最终促使我们启动了制品仓库的平替与迁移工作。

------

## **我们遇到的问题**

我们的 Nexus `3.78.2-04` 底层数据库使用的是 PostgreSQL，不是默认的 OrientDB。在存储层面，PostgreSQL 确实比 OrientDB 更稳定、更易维护。但实际运行下来发现：

> **真正卡住我们的是 Community Edition 的用量上限，跟数据库选型关系不大。**

### **1. 组件数量上限与请求上限是“硬性封顶”**

Nexus Community Edition 对使用量有严格限制：

- **最多 100,000 个 components**
- **每天最多 200,000 次 requests**

这两个限制与数据库类型无关，即使使用 PostgreSQL，也无法突破。

我们内部的实际情况是：

- 仓库数量 20+
- 组件数量超过 80,000
- 日均请求量约 100,000
- 高峰期会触发 200,000 请求上限

一旦触发上限：

> **新的制品无法上传，构建流水线直接卡住，这是一个致命的问题。**

这对一个 500 多人的研发团队来说影响极大，尤其是多条业务线并行发布时，构建中断会造成明显的交付风险。

### **2. 数据库选型不是瓶颈所在**

我们一开始就在用 PostgreSQL，存储可靠性和运维便利性没有问题。但 PostgreSQL 解决不了以下问题：

- 100,000 个 components 的数量上限——跟用什么数据库无关
- 200,000 次 requests 的日请求上限——同上
- 多副本部署、滚动升级、弹性伸缩——这些是 Nexus 自身架构不支持的

换句话说，Nexus 的瓶颈不在数据库，而在 Community Edition 本身的用量限制和部署架构。

### **3. 架构限制导致运维成本持续升高**

即使使用 PostgreSQL，Nexus `3.78.2-04` Community Edition 仍然：

- 不支持多副本部署
- 不支持滚动升级，每次升级都需要停服
- 不支持弹性伸缩
- 高可用架构难以实现

------

## **为什么选择 kkRepo**

kkRepo 项目地址：https://github.com/klboke/kkRepo

- 我们也看过其他制品仓库方案，但最终选择 kkRepo，主要原因有两个：

> **kkRepo 不存在组件数量上限，也不存在请求上限。**

这意味着：

- 不会因为业务增长而触发“硬性封顶”
- 不会因为高峰期请求量过大而导致构建中断
- 不需要为解除限制而购买昂贵的商业版
- 不需要担心未来规模扩张带来的架构瓶颈

同时，kkRepo 完全兼容 Nexus 的客户端协议和 URL 布局，使得迁移成本极低：

- CI 配置无需修改
- Maven settings.xml 无需修改
- npm registry 无需修改
- Docker registry 无需修改

对于一个 500 多人的研发团队来说，不用逐个修改 CI 配置，迁移成本大大降低。

### **kkRepo 的技术特点**

kkRepo 在架构上做了几个对我们比较关键的改进：

| 对比项 | Nexus `3.78.2-04` | kkRepo |
| --- | --- | --- |
| 元数据存储 | PostgreSQL | MySQL |
| Blob 存储 | OSS/file/S3 | OSS/S3 兼容 |
| 多副本部署 | 不支持 | 支持 |
| 滚动升级 | 不支持 | 支持 |
| 弹性伸缩 | 不支持 | 支持 |
| 客户端兼容 | - | 兼容 Nexus URL 和协议 |
| 迁移工具 | - | 内置迁移工具 |

## **迁移过程**

整个迁移过程分为四个阶段，整体非常平滑。

### **1. 环境准备**

我们通过 Docker Compose 部署 kkRepo：

- 连接生产 MySQL 实例
- 配置 OSS blob store
- 配置多副本部署能力（为未来扩展做准备）

kkRepo 的部署过程比较简洁，Docker Compose 拉起来就能用。

### **2. 元数据迁移**

在 kkRepo 管理后台 `/admin/` 的 Nexus Metadata 页面执行：

1. **Run preflight**

   - 检查源 Nexus 的配置与兼容性
   - 主要阻塞点是：源 Nexus 必须开启 Script REST API

2. **处理阻塞项**
    在源 Nexus 的 nexus.properties 中添加：

   ```
   nexus.scripts.allowCreation=true
   ```

   修改后重启 Nexus。

3. **Run migration**
    迁移以下内容：

   - 用户
   - 角色
   - 权限
   - blob store
   - 仓库定义

### **3. 仓库数据迁移**

在 kkRepo 的 Nexus Repository Data 页面执行：

1. **Sync metadata**
    扫描源 Nexus hosted 仓库的资产元数据。
2. **Sync packages**
    迁移真实的 blob/package 数据到 kkRepo。

对于数据量较大的仓库，我们把并发度从默认的 8 调到 32，迁移速度明显加快。

### **4. 增量同步与切换**

正式切换前，我们执行了一轮增量迁移，确保迁移窗口内新增的数据也同步到 kkRepo。

确认无问题后，通过 DNS 将原 Nexus 域名指向 kkRepo，实现无感切换。

------

## **迁移注意事项（非常关键）**

结合我们的实践经验，有几个需要特别注意的点：

1. **源端必须开启 Script API**
    否则迁移工具无法读取部分数据。
2. **确保 Nexus 的 blob store 配置一致**
    否则迁移时会出现路径不一致的问题。
3. **迁移期间尽量减少构建任务**
    虽然支持增量迁移，但减少写入能降低风险。
4. **并发度可调，但要注意源端压力**
    并发度过高可能导致源 Nexus 负载过高。
5. **DNS 切换前务必做一次增量同步**
    确保迁移窗口内的数据一致性。
6. **CI 无感迁移是最大优势**
    迁移后无需修改任何构建脚本或客户端配置。

------

## **迁移效果与收益**

迁移到 kkRepo 后，实际效果：

- 不再有组件数量和请求量上限，不用担心触发硬性封顶
- 支持多副本部署和滚动升级，升级不再需要停服
- 可以按需扩容，支持弹性伸缩
- 元数据在 MySQL 中，blob 在 OSS 中，数据备份和恢复更方便
- CI 配置和客户端全部无需修改，切换后业务侧没有收到任何失败反馈

对我们来说，最直接的改善是：构建流水线不会再因为触发请求上限而中断了。
